"Downloads のデータをホームディレクトリの別ディレクトリに移動とかできないの？
STEP 1 · 試す
v3 で実行 → 失敗
scan ~/Downloads & suggest --target ~/Documents で apply まで通そうとした。
エラー: 存在しないファイル: photo.png
STEP 2 · 原因
plan の from が相対パス
単一 root は無条件で in-place 扱い → from は scan 元への相対。apply は target からの相対と解釈してファイルを見つけられない。
"from": "photo.png" ← どこの photo.png?
STEP 3 · 修正
cross_target 判定を追加
multi_root OR target≠scan_root のとき consolidate モードで動作。from は絶対パスで出力。
"from": "/Users/.../Downloads/photo.png" ✓
STEP 4 · v4 で動く
scan ~/Downloads → suggest --target ~/Documents → preview → apply
Downloads の中身が ~/Documents/コード/、~/Documents/画像/、~/Documents/ドキュメント/メモ/ などに振り分けられる。全部 manifest に記録、undo で完全に元の Downloads に戻る。

v3 のキモ: 依存解析クラスタ

import を読むから、
関係するファイルを切り離さない。

scan が各コードファイルを開いて import / require / src= / @import を抽出 → 連結したファイル群を1つのクラスタとして scan.json に記録。suggest はクラスタを尊重して同じ subfolder に配置する。手で plan を変えてクラスタを分断すると preview が止める。

    v4 で動くもの
    
ユーザの一言に、
本体で答える。

    

      
01 v4 NEW
Downloads → ホーム配下への振り分け
「~/Downloads を ~/Documents の下に振り分けて」が本当に動く。scan の対象と target dir が違うことが自然に検出され、plan の from は絶対パス、target 配下のファイルは自動 skip。

      
02 v4 NEW
cross_target 判定の一般化
multi_root だけでなく 「単一 root でも target が異なる場合」もまとめて consolidate モードとして扱う。境界条件を1か所に集約してテストで保証。

      
03 v3〜
依存解析クラスタ
Python の import、JS の require / import、HTML の src= / href=、CSS の @import を読む。連結したファイル群は同じ subfolder へまとめて配置。

      
04 v3〜
suggest / verify / redo / Ctrl-C 安全終了
scan から plan の自動生成、apply 後の整合性チェック、undo の取り消し、中断時の journal フラッシュ。「やり直し」のすべての方向に答える。

      
05
削除しない隔離（v2〜）
不要・重複ファイルはゴミ箱ではなく _捨て/ へ。assess で危険な対象は止める。

      
06
集約＋重複排除（v2〜）
複数の場所に散らばった資料を1か所に集め、同一内容（sha1一致）の重複を1つに絞る。
